# نظام السلف النقدية (Naif Loan System)

نظام داخلي مبني بـ **Next.js 14 (App Router)** و **Prisma 5** و **PostgreSQL (Supabase)** لإدارة طلبات السلف النقدية المؤقتة الخاصة بالمهمات (نموذج ١٨) وتسوياتها (نموذج ١٩)، من تقديم الطلب وحتى الاعتماد النهائي، مع تتبّع آلي لمواعيد التسوية وتنبيهات بريدية.

> هذا الملف موجّه للفنيين/المطورين الذين سيتولون تشغيل أو صيانة أو نقل المنصة، بما يشمل احتمال نقلها إلى بيئة استضافة الجامعة الرسمية.

---

## 1. نظرة عامة على الفكرة

- **الموظف** يقدّم طلب سلفة (نموذج ١٨) لتغطية مصاريف مهمة (سفر/تدريب/نشاط)، مرفقاً ببنود الصرف المتوقعة وتصنيف الوجهة.
- **المراجع (REVIEWER)** يراجع الطلب: معاينة، تعديل البنود، إعادته للموظف، أو اعتماده.
- بعد انتهاء المهمة، يقدّم الموظف **تسوية السلفة** (نموذج ١٩) ببيانات الصرف الفعلي (مؤيد بمستندات / غير مؤيد / الوفر أو الزيادة).
- النظام يحسب **مهلة التسوية** تلقائياً بناءً على تاريخ نهاية المهمة وتصنيف الوجهة (محلي/عربي/أوروبا ومغرب عربي/أمريكا وآسيا البعيدة)، ويُصنّف كل سلفة مفتوحة حسب قربها من المهلة (ضمن المهلة / قاربت / وشيكة / متأخرة).
- مهمة مجدولة (cron) تفحص يومياً السلف المتأخرة وترسل تنبيهات بالبريد الإلكتروني للموظف والإدارة.
- يمكن ربط أي سلفة بدورة تدريبية (`courseId`) من **منصة إقفال الدورات التدريبية**؛ عند طباعة نموذج ١٨ أو نموذج ١٩ يُرسل النظام Webhook يحدّث حالة عنصر الإقفال المقابل تلقائياً في المنصة الأخرى (انظر القسم 9).

## 3. الأدوار والصلاحيات

تُدار الأدوار في [`lib/auth.ts`](lib/auth.ts) عبر النوع:

```ts
type SessionRole = 'EMPLOYEE' | 'ADMIN' | 'REVIEWER'
```

- **EMPLOYEE (موظف)**: يقدّم طلبات السلف، يتابع حالتها، يقدّم التسويات، ويمكنه طلب "إعادة فتح" معاملة معتمدة/مسوّاة لتعديلها (مع كتابة سبب).
- **REVIEWER (مراجع)**: يرى قائمة الطلبات المرفوعة، يعاين/يعدّل بنود الطلب، يعيده للموظف مع ملاحظة، أو يعتمده (يتحول لون البطاقة لأخضر مع شارة "✓ طلب معتمد"). كذلك يعتمد التسويات (نموذج ١٩) ويقرر بشأن طلبات "إعادة الفتح".
- **ADMIN (مدير)**: صلاحيات كاملة — إدارة المستخدمين والأدوار، إعدادات النظام، عداد الأرقام المرجعية، التقارير، البريد الجماعي. مستخدم المدير الأعلى (Super Admin) مُعرّف بثابت `SUPER_ADMIN_EMAIL = 'od@nauss.edu.sa'` في `lib/auth.ts`.

يدعم النظام أدواراً متعددة لمستخدم واحد (`roles: Json` في جدول `users`)، عبر دوال مساعدة: `normalizeRoles`, `getPrimaryRole`, `hasRole`, `canManageAllLoans`, `isSuperAdmin`.

## 4. نموذج البيانات (Prisma Schema)

أهم الجداول في [`prisma/schema.prisma`](prisma/schema.prisma):

- **`User`** (`users`): الحساب الموحّد — `role` الأساسي + `roles` (مصفوفة JSON)، `signatureImage` (التوقيع الإلكتروني المستخدم في النماذج)، وربط اختياري بمنصة الإقفال عبر `closureSystemUserId` (نفس البريد الإلكتروني = نفس المستخدم في المنصتين).
- **`Loan`** (`loans`): طلب السلفة (نموذج ١٨) — بيانات النشاط/الوجهة/المبلغ/التواريخ، `destinationCategory` (تصنيف الوجهة)، `settlementDeadline` (مهلة التسوية المحسوبة تلقائياً)، `reviewStatus` (PENDING/REVIEWED/RETURNED)، `isSettled` و `settlementStatus` (NOT_STARTED/IN_PROGRESS/SUBMITTED/APPROVED/OVERDUE)، حقول طلب إعادة الفتح (`recallRequested`, `recallReason`, `recallRequestedAt`)، وحقول الربط بمنصة الإقفال (`courseId`, `courseCode`).
- **`LoanItem`** (`loan_items`): بنود الصرف المتوقعة لكل سلفة (فئة + مبلغ).
- **`Settlement`** (`settlements`): تسوية السلفة (نموذج ١٩) — مؤيد/غير مؤيد/الإجمالي/الوفر/الزيادة، وملف JSON للفواتير والمرفقات.
- **`LoanNotification`** (`loan_notifications`): إشعارات داخلية للمستخدمين (`NotificationType`: LOAN_CREATED, LOAN_REVIEWED, LOAN_RETURNED, SETTLEMENT_REMINDER, SETTLEMENT_OVERDUE, SETTLEMENT_APPROVED, SYSTEM).
- **`LoanAlert`** (`loan_alerts`): سجل تنبيهات البريد المُرسلة (لتجنّب التكرار) — REMINDER_3DAYS / OVERDUE / SEVERE_OVERDUE.
- **`LoanSequence`** (`loan_sequence`): عداد ذرّي (atomic) لتوليد الرقم المرجعي بصيغة `وت/26/0001` دون تعارض عند الطلبات المتزامنة.

### حساب مهلة التسوية (`lib/settlement-deadline.ts`)

تصنيفات الوجهة (`DestinationCategory`) وتأثيرها على عدد أيام العمل المضافة قبل/بعد المهمة:

| التصنيف | أيام قبل/بعد | أمثلة |
|---|---|---|
| `DOMESTIC` | 1 / 1 | الرياض، جدة، الدمام... |
| `ARAB` | 2 / 2 | الإمارات، مصر، الأردن... |
| `EUROPE_MAGHREB` | 3 / 3 | فرنسا، ألمانيا، الجزائر... |
| `AMERICAS_FAR` | 4 / 4 | أمريكا، كندا، أستراليا، اليابان... |

`settlementDeadline = endDate + (أيام بعد التصنيف) + 10 أيام عمل` (تُحسب باستثناء الجمعة والسبت عبر `addWorkingDays`/`isWorkingDay`).

حالة كل سلفة مفتوحة (`getDeadlineStatus`) تُصنَّف إلى: `safe` (ضمن المهلة)، `warning` (قاربت المهلة)، `critical` (المهلة وشيكة، يوم أو أقل)، `overdue` (تجاوزت المهلة) — وتُستخدم لتلوين البطاقات والعدادات في لوحة المعلومات.

## 6. الميزات الرئيسية

- **طلب السلفة (نموذج ١٨)**: تعبئة بيانات المهمة والوجهة والبنود المتوقعة، توليد رقم مرجعي تلقائي، طباعة النموذج.
- **قائمة المراجعة (Reviewer Queue)**: تعرض "المعاملات النشطة" فقط (تستثني السلف المسوّاة/المؤرشفة)، مع أزرار واضحة لكل طلب: معاينة، تعديل البنود (قبل الاعتماد فقط)، إعادة للموظف، اعتماد/إلغاء الاعتماد. عند الاعتماد تتحول البطاقة إلى خلفية خضراء مع شارة "✓ طلب معتمد".
- **تسوية السلفة (نموذج ١٩)**: إدخال المصروف الفعلي (مؤيد/غير مؤيد)، حساب الوفر أو الزيادة تلقائياً، مراجعة واعتماد من المراجع.
- **طلب إعادة فتح معاملة**: يسمح للموظف بطلب إعادة فتح معاملة معتمدة أو مسوّاة (مع سبب)، وللمراجع قبول أو رفض الطلب.
- **عداد مهلة التسوية**: عداد لوني واحد (أخضر/أحمر) لكل سلفة مفتوحة يوضح الأيام المتبقية أو أيام التأخير (`getSettlementCountdown` في `DashboardClient.tsx`).
- **لوحة المعلومات (Dashboard)**:
  - عدادات تسوية السلف المفتوحة (بطاقات عمودية بتصميم رقمي كبير، أخضر للمتبقي/أحمر للمتأخر).
  - رسم بياني Radar لـ"درجة استعجال التسوية" لكل موظف (يرتفع مع اقتراب/تجاوز الموعد).
  - رسوم بيانية إضافية: المبالغ الشهرية (مطلوب/مسوّى)، توزيع حالات الطلبات، استخدام بنود الصرف حسب الفئة، وأكثر/أقل البنود استخدامًا.
- **الإشعارات**: إشعارات داخلية (`LoanNotification`) + بريد إلكتروني عبر Resend عند إنشاء/مراجعة/إعادة/اعتماد السلفة أو التسوية، وعند طلبات إعادة الفتح وقراراتها.
- **التنبيهات الآلية للمهلات**: مهمة Cron يومية (`/api/cron/check-deadlines`) تفحص السلف القاربة على الانتهاء أو المتأخرة وترسل تنبيهات (REMINDER_3DAYS / OVERDUE / SEVERE_OVERDUE) مع تسجيلها في `LoanAlert` لمنع تكرار الإرسال لنفس السلفة ونوع التنبيه.
- **الطباعة والتصدير**: توليد مستندات Word لنموذج ١٨/١٩ وتقرير الجهات عبر `docxtemplater`.
- **لوحة المدير**: إدارة المستخدمين والأدوار، إعدادات النظام (`system-settings.ts`)، عداد الأرقام المرجعية (`/api/admin/sequence`)، بريد جماعي وتجربة البريد، تنبيهات يدوية.
- **تقارير**: تقرير استخدام بنود الصرف (`/api/reports/item-usage`) لتحليل الفئات الأكثر/الأقل استخداماً.

## 7. التشغيل المحلي

1. انسخ ملف البيئة:
   ```
   cp .env.example .env
   ```
2. عدّل المتغيرات في `.env` (انظر جدول متغيرات البيئة في القسم 8).
3. ثبّت الحزم:
   ```
   npm install
   ```
4. زامن مخطط قاعدة البيانات (يُنفَّذ تلقائياً أيضاً ضمن `postinstall: prisma generate`):
   ```
   npx prisma db push
   ```
5. شغّل المشروع:
   ```
   npm run dev
   ```

### أوامر مفيدة

| الأمر | الوصف |
|---|---|
| `npm run dev` | تشغيل بيئة التطوير |
| `npm run build` | بناء المشروع للإنتاج |
| `npm run start` | تشغيل النسخة المبنية |
| `npm run resync:closure` | إعادة مزامنة عناصر منصة إقفال الدورات يدوياً (`scripts/resync-closure-elements.mjs`) |
| `npx prisma db push` | مزامنة مخطط Prisma مع قاعدة البيانات (بدون migration history) |
| `npx prisma studio` | تصفح/تعديل البيانات عبر واجهة Prisma Studio |

## 8. النشر على Vercel ومتغيرات البيئة

### خطوات النشر

1. ارفع المشروع إلى GitHub (المستودع: `nauss-nif/solf`، الفرع: `main`).
2. أنشئ مشروعاً جديداً في Vercel واربط المستودع.
3. أضف متغيرات البيئة من الجدول أدناه في إعدادات المشروع على Vercel.
4. تأكد أن قاعدة بيانات Supabase في منطقة قريبة جغرافياً من منطقة استضافة Vercel (حالياً `bom1`) لتقليل زمن الاستجابة.
5. بعد أول نشر، نفّذ مزامنة المخطط مع قاعدة البيانات من بيئة آمنة (محلياً أو عبر سكربت CI منفصل):
   ```
   npx prisma db push
   ```
6. **لا تضع `prisma db push` داخل أمر `build`** على Vercel — لتجنّب تعديل قاعدة البيانات تلقائياً أثناء كل عملية نشر.

### متغيرات البيئة

| المتغير | الوصف | مطلوب؟ |
|---|---|---|

## 9. المهام المجدولة (Cron)

يُعرَّف في [`vercel.json`](vercel.json):

```json
{
  "regions": ["bom1"],
  "crons": [
    { "path": "/api/cron/check-deadlines", "schedule": "0 6 * * 0-4" }
  ]
}
```

- يعمل من الأحد إلى الخميس (`0-4`) عند الساعة 6:00 صباحاً (بتوقيت UTC حسب جدولة Vercel).
- يفحص جميع السلف المفتوحة، يصنّف حالتها حسب `getDeadlineStatus` (ضمن المهلة/قاربت/وشيكة/متأخرة)، ويرسل تنبيهات بريدية (REMINDER_3DAYS, OVERDUE, SEVERE_OVERDUE) مع تسجيلها في `LoanAlert` لمنع تكرار الإرسال لنفس السلفة ونوع التنبيه.
- نقطة النهاية محمية بـ `CRON_SECRET` — أي استدعاء يدوي لها يجب أن يحمل هذا المفتاح.

## 10. التكامل مع منصة إقفال الدورات التدريبية

عند ربط سلفة بدورة تدريبية (`Loan.courseId`)، يقوم النظام بإرسال إشعار Webhook إلى منصة إقفال الدورات عند طباعة نموذج ١٨ (نوع `advance_req`) أو نموذج ١٩ (نوع `settlement`)، عبر [`lib/closure-integration.ts`](lib/closure-integration.ts):

- **الوجهة**: `CLOSURE_WEBHOOK_URL` على منصة الإقفال — يستقبلها المسار `pages/api/webhooks/solf-closure-sync.js` في مشروع الإقفال (`C:\nif\end`).
- **المصادقة**: رأس `Authorization: Bearer <CLOSURE_WEBHOOK_SECRET>` (يجب أن يطابق `WEBHOOK_SECRET`/`SOLF_WEBHOOK_SECRET` في منصة الإقفال).
- **محتوى الطلب**: `{ type, courseId, employeeEmail, referenceNumber, amount, advanceAmount, spentAmount, printedAt, solfLoanId }`.
- **الأثر على الطرف الآخر**: منصة الإقفال تبحث عن عنصر إقفال (`CourseClosureTracking`) مطابق للدورة ونوع العنصر (`advance_req` أو `settlement`) وتُحدّث حالته تلقائياً.
- **إذا لم تكن المتغيرات مهيأة** (`CLOSURE_WEBHOOK_URL`/`CLOSURE_WEBHOOK_SECRET` غير موجودة)، أو لم تكن السلفة مرتبطة بدورة (`courseId` فارغ)، تُتجاهل المزامنة بصمت (`skipped: true`) دون أن تفشل عملية الطباعة.
- **إعادة المزامنة اليدوية**: عند الحاجة لإعادة إرسال عناصر سابقة، استخدم `npm run resync:closure` (`scripts/resync-closure-elements.mjs`).
- **ربط الحسابات**: حقل `User.closureSystemUserId` (اختياري) يسمح بربط مستخدم منصة السلف بمستخدم مطابق في منصة الإقفال؛ المطابقة الأساسية تتم عبر البريد الإلكتروني (نفس البريد = نفس الشخص في المنصتين).

## 11. ملاحظات أمنية وعملية

- ملف `.env` غير مرفوع داخل المستودع — استخدم `.env.example` كقالب فقط.
- يجب تدوير (تغيير) أي بيانات اتصال بقاعدة البيانات أو مفاتيح API سبق رفعها داخل المستودع أو مشاركتها خارجياً.
- نقاط API الحساسة (تسجيل الدخول، إعادة تعيين كلمة المرور) محمية بتحديد معدل الطلبات عبر `lib/rate-limit.ts`.
- التوقيع الإلكتروني للمستخدمين (`User.signatureImage`) يُستخدم في توليد نماذج الطباعة المعتمدة — يجب التعامل معه كبيانات حساسة عند النقل أو النسخ الاحتياطي.
- عند نقل المنصة إلى بيئة الجامعة الرسمية:
  1. أنشئ قاعدة بيانات PostgreSQL جديدة وحدّث `DATABASE_URL` و `DIRECT_URL`.
  2. نفّذ `npx prisma db push` لإنشاء المخطط (الجداول والأنواع المذكورة في القسم 4).
  3. أعد ضبط جميع متغيرات البيئة في القسم 8 على بيئة الاستضافة الجديدة، مع تدوير كل المفاتيح السرية (`AUTH_SECRET`, `CRON_SECRET`, `CLOSURE_WEBHOOK_SECRET`, `RESEND_API_KEY`).
  4. إن استمر التكامل مع منصة إقفال الدورات، حدّث `CLOSURE_WEBHOOK_URL` ليشير إلى عنوان منصة الإقفال الجديد، وتأكد من تطابق `CLOSURE_WEBHOOK_SECRET`/`WEBHOOK_SECRET` بين المنصتين.
  5. حدّث جدولة Cron حسب قدرات بيئة الاستضافة الجديدة إذا لم تكن Vercel Cron متاحة (يمكن استبدالها بأي مُجدول خارجي يستدعي `/api/cron/check-deadlines` مع رأس `CRON_SECRET`).
